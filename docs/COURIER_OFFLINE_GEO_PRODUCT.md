# Курьерское приложение: offline-first, Москва, гео — продукт и архитектура

Документ для **собеседования** и для команды: что заявляем на уровне продукта, как это согласуется с ограничениями (сеть, глушилки), что уже сделано в репозитории и что логично добить дальше.

---

## Короткий «идеальный ответ» (2–3 абзаца)

**Задача** — курьер не должен зависеть от стабильного интернета и идеального GPS: в центре Москвы и отдельных зонах бывают «чёрные дыры» по LTE, белые списки операторов и помехи спутниковым сигналам.

**Архитектура** — мобильный клиент курьера в парадигме **offline-first**: операции и трек **сначала попадают в локальное хранилище** и не блокируют работу курьера; синхронизация с бэкендом **асинхронная**, при любой доступной связи. Для координат ведётся **журнал трека** (время, lat/lon, источник, точность); на сервер уходит **пакетами** (батчами ограниченного размера). **Владелец подтверждённых статусов заказа — сервер**: при конфликте клиент **ребазирует** локальные «черновики» под факт с сервера, а не перетирает подтверждённые состояния.

**UX и бизнес** — у курьера явные состояния **Online / Offline / ухудшение GPS**; у клиента и диспетчера при обрыве связи показываем **последний известный апдейт** и честный текст вроде «данные обновятся при восстановлении сети», плюс инструменты эскалации (звонок, правила «нет координат N минут»). После восстановления связи сервер получает **полный трек задним числом** из локального журнала — можно считать SLA, разбирать споры и строить метрики. В зонах глушения опираемся на **грубое позиционирование** (LBS / Wi‑Fi там, где ОС его отдаёт) и **фильтрацию скачков** на клиенте и сервере, чтобы не показывать «телепортации» на карте.

## 1. Offline-first (детально)

| Принцип | Смысл |
|--------|--------|
| Сеть «по умолчанию ненадёжна» | UI и сценарии не предполагают постоянный REST. |
| Локальная очередь / журнал | Все критичные действия дублируются локально до ack с сервера. |
| Батчи | Трек отправляется пачками (в продукте часто **20–50** точек за запрос; верхний предел на бэке остаётся защитой). |
| Конфликты | Сервер — источник правды для **подтверждённых** статусов; клиент хранит `serverVersion` / `syncedAt` и при рассинхроне **мерджит**, а не затирает серверное. |

### Что есть в этом репозитории

- Журнал **очереди координат** на диске (`LocationQueueStore`), отправка **`POST .../location/batch`**.
- Отправка **чанками** (например, по 40 точек за вызов), пока очередь не опустеет или не упёрлись в ошибку сети.
- Бэкенд: сортировка по времени, прокидка в **geo** по порядку (если задан `GEO_SERVICE_URL`), в Postgres — последняя точка.

### Что типично добавляют дальше (outbox)

- **SQLite / CoreData / Realm**: единая **outbox-таблица** для статусов, комментариев, «принял заказ» с идемпотентными `client_op_id`.
- Воркер синка: exponential backoff, приоритет критичных операций.
- Ручная выдача заказа клиенту: локальный экран + **QR / одноразовый код**, который сервер потом валидирует (у нас уже есть QR-поток прибытия на кухню — аналогичный паттерн).

## 2. Отключение / ограничение интернета (Москва)

**Сценарии:** нет мобильного интернета, белые списки, нестабильный LTE в центре.

**Поведение:**

- Курьер: заранее загруженные задачи и маршрут (кэш), работа в офлайне, накопление событий.
- Клиент: последняя позиция + «связь временно отсутствует».
- Диспетчер: последняя валидная точка, таймер «нет апдейтов N минут → эскалация», ручные каналы.

### Что есть в репозитории

- Индикатор **сети** (NWPathMonitor): онлайн / ограничено / офлайн в UI курьера.
- Очередь точек при офлайне + flush при появлении сети и при pull-to-refresh.

## 3. Глушилки и деградация GPS

**Факты для собеседования:** в отдельных зонах возможны помехи GNSS; в туннелях/развязках навигация деградирует; разумный fallback — **сигналы от ОС** (часто грубые: вышки, Wi‑Fi).

**Стратегия:**

- Помечать **источник** и **точность** (`horizontalAccuracy` на iOS).
- На бэке (geo) хранить `source`, опционально `accuracy_m`; правила «телепорт» уже в духе speed-check (см. geo-сервис).
- Дальше: явный порог «не показывать клиенту точку с accuracy > X м» или флаг `approximate` в API для карты.

### Что есть в репозитории

- В батч передаются **`source`** (в т.ч. эвристика `gps` / `gps_degraded_or_wifi_lbs` / `significant_cell` и т.д.) и **`accuracy_m`**.
- Geo-сервис сохраняет **`accuracy_m`** в документе точки (если передан).

### Что добавить позже

- Калман / сглаживание на клиенте.
- Явная политика UI для клиента: «последнее обновление N минут назад», не прятать курьера, а показывать неопределённость.

## 4. Поведение глазами пользователей

| Роль | Норма | Проблемы |
|------|--------|----------|
| Курьер | Online, GPS ok | Offline / constrained; **GPS degraded** по точности |
| Клиент | Плавный трек | Статичная последняя точка + время апдейта + честный текст |
| Бизнес | Операционный контроль | После синка — полный трек для SLA и разборов |
