# Диспатчер (dispatcher)

Это учебно‑боевой проект: микросервисы на FastAPI, фронт на React, и iOS‑приложение курьера.  
Главная идея простая: **кухня готовит**, **диспатчер назначает**, **курьер везёт**, **офис смотрит правду в логах**.

Если коротко по фичам:

- **Кухня (web)**: очередь заказов, создание заказа, ручное назначение курьера.
- **Офис (web)**: таймлайн решений, детали, аналитика override‑ов (когда люди победили алгоритм).
- **Курьер (iOS)**: смена, статусы, текущие заказы, геопозиция авто‑отправкой, QR‑подтверждение “я реально на кухне”.
- **Сервисы**: Order / Courier / Dispatcher / Log (+ Geo / Config / Notification / Gateway3PL как окружение).

Да, это то самое место, где “быстро поправить одну строчку” внезапно превращается в `docker-compose build`. Это нормально.

## Структура репозитория (куда смотреть, когда всё горит)

- `**frontend/`** — React + Vite + MUI (веб: Кухня и Офис)
- `**testMnogoIOS/**` — iOS‑приложение курьера (SwiftUI)
- `**services/**` — микросервисы FastAPI:
  - `**order/**` — заказы, статусы, ручное назначение, WebSocket `/ws` для кухни
  - `**courier/**` — курьеры, смены, статусы, гео, WebSocket `/ws`, QR‑прибытие на кухню
  - `**dispatcher/**` — воркер: слушает события и делает “умное назначение” (или только логирует в shadow mode)
  - `**log/**` — Decision Log (ClickHouse): решения и аналитика
  - (и ещё `**geo/**`, `**config/**`, `**notification/**`, `**gateway3pl/**` — как в `docker-compose.yml`)
- `**pkg/**` — общий код/модели (не путать с “пакетом с пакетами”)
- `**docker-compose.yml**` — наш “оркестр из контейнеров”, дирижёр — Docker


Открывай `http://localhost:3000`.

Веб ходит к бэкендам через прокси Vite по путям:

- `/api/order/*`
- `/api/courier/*`
- `/api/log/*`

### 3) Открыть iOS‑приложение курьера

Проект: `testMnogoIOS/testMnogoIOS.xcodeproj`.

В приложении вводишь:

- **URL Courier API** (например `http://192.168.1.84:8001` на реальном устройстве)

Важно: на реальном устройстве нельзя писать `localhost` — это “сам телефон”, а не твой Mac.

## Как пользоваться (короткий сценарий)

### Кухня

- **Кухня → Создать заказ**: создаёшь заказ для `kitchen_id`.
- **Кухня → Очередь заказов**: видишь список, можешь вручную назначить курьера.
- **Кухня → Курьеры**: список курьеров кухни + **QR прибытия** (если курьер нажал “Я на кухне”).

### Курьер (iOS)

- **Начать смену** → статус становится `idle`.
- Когда назначили заказ → статус `delivering`, заказ появляется в “Текущие заказы”.
- После доставки → “Доставлен”.
- Если курьер возвращается → ставит `returning`.
- В статусе “Возвращаюсь” появляется кнопка **«Я на кухне»**:
  - на кухне появляется QR (персональный, с именем курьера),
  - курьер сканирует QR камерой,
  - статус становится `idle`, QR исчезает.

QR живёт **15 секунд**, потом при новом нажатии генерируется заново.

### Офис

- **Офис → Решения**: последние решения диспатчера (и ручные override‑ы).
- **Офис → Аналитика**: доля ручных переопределений.

## Примечания (чтобы не наступать на грабли)

- **Камера на симуляторе iOS**: не работает. QR‑сканер — только на реальном устройстве.
- **NSCameraUsageDescription**: добавлен в проект, иначе iOS справедливо падает.
- **Лог решений (ClickHouse)**: поле `assigned_to` хранится как `String` (потому что курьеры — 6‑значные логины).
